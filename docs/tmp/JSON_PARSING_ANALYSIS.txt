## 2. UNHANDLED EDGE CASES (MISSING/FRAGILE SHAPES)

### 2.1 Streaming Incomplete JSON
**Problem:** When LLM returns JSON via streaming, chunks arrive incomplete
**Test case:** `{"quizzes": [{"question": "Q1", ...` (missing closing braces)
**Current behavior:** _extract_quiz_json_payload() returns NONE
**Impact:** Chain.invoke() throws exception → caught and re-raised if repaired is None
**Result:** OUTPUT_PARSING_FAILURE

### 2.2 Trailing Comma in JSON
**Problem:** LLM generates JSON with trailing comma
**Test case:** `{"quizzes": [{...},]}`
**Current behavior:** json.loads() fails → returns NONE
**Impact:** Not recovered by fallback parser

### 2.3 Empty Required Arrays → Validation Failure
**Problem:** Empty "quizzes" array passes JSON parse but fails Pydantic
**Current behavior:** Result = {"quizzes": []} → model_validate OK
**But:** If "quizzes" key is MISSING: model_validate fails with validation error
**Impact:** Empty object `{}` → JSON parse OK but Pydantic rejects it

### 2.4 Type Mismatches in Nested Objects
**Problem:** LLM generates wrong types for required fields
**Examples:**
- options: "ABCD" instead of ["A","B","C","D"]
- timestamp: 120 instead of "00:02:00"
- video_url: {url: "..."} instead of "..."
**Current behavior:** JSON parse OK → Pydantic validation fails silently
**Impact:** Falls through to error handler without hint

### 2.5 Array Instead of Object
**Problem:** LLM returns `[{...}]` instead of `{"quizzes": [...]}`
**Test case:** `[{"question": "Q1", ...}]`
**Current behavior:** Regex matches `[...]` → json.loads() succeeds but model_validate fails
**Impact:** Accepted by parser, rejected by Pydantic

### 2.6 Supervisor Tool Call Shape Mismatches
**Fragile case:** args/arguments is not a dict
**Example:** `{"name": "GenerateQuiz", "args": "not_a_dict"}`
**Current behavior:** Normalized to `{"name": "...", "args": {}}` (coerced to empty dict)
**Impact:** Query/topic lost silently

### 2.7 Multiple JSON Objects in Text
**Problem:** LLM generates multiple JSON objects
**Example:** `{...quiz1...} {... quiz2...}`
**Current behavior:** Regex captures first `{...}` only
**Impact:** Second object ignored without warning

### 2.8 Non-Matching Array Lengths in VideoAnswer
**Problem:** Tutor returns mismatched arrays
**Example:** 
```json
{
  "text": "...",
  "video_url": ["url1", "url2"],
  "filename": ["file1"],  # MISMATCH!
  ...
}
```
**Current behavior:** JSON parse OK → But violates documented constraint
**Impact:** Frontend rendering breaks when accessing by index

### 2.9 Unescaped Newlines in JSON Strings
**Problem:** LLM embeds literal newline in JSON string value
**Example:** `{"text": "line1\nline2"}` where \n is actual newline
**Current behavior:** json.loads() fails
**Impact:** Silently rejected

### 2.10 Markdown Code Block Typos
**Problem:** Case sensitivity or extra spaces in fence
**Examples:** ` ```Json ` (capital J), ` ``` json ` (space before json)
**Current behavior:** _extract_quiz_json_payload() regex is case-insensitive but may miss
**Impact:** Falls back to bare JSON extraction


## 3. FALLBACK PARSER ROBUSTNESS

### 3.1 _extract_quiz_json_payload() Analysis (lines 43-64)
```python
def _extract_quiz_json_payload(raw: str):
    if not isinstance(raw, str):
        return None  # ISSUE: type error silently returns None
    
    text = raw.strip()
    fenced_match = re.search(r"```json\s*(\{[\s\S]*?\})\s*```", text, re.IGNORECASE)
    if fenced_match:
        text = fenced_match.group(1).strip()
    
    try:
        return json.loads(text)
    except Exception:
        pass  # Silent fail - no logging
    
    obj_match = re.search(r"(\{[\s\S]*\})", text)
    if not obj_match:
        return None
    
    try:
        return json.loads(obj_match.group(1))
    except Exception:
        return None  # Silent fail
```

**Issues with fallback parser:**
1. Greedy regex: `(\{[\s\S]*\})` matches FIRST `{` to LAST `}`
   - Risk: If text contains multiple JSON objects, includes all
   - Example: `{obj1} some text {obj2}` → extracts `{obj1} some text {obj2}` → fails
   
2. No validation after extraction
   - Accepts malformed but-technically-parseable JSON
   - Example: `{"quizzes": [{"question": "Q1"}]}` (missing fields) passes extraction
   - Pydantic catches it but error info is lost

3. Silent exceptions
   - No logging of what failed
   - Makes debugging streaming issues hard

## 6. ROOT CAUSE HYPOTHESIS FOR STREAMING INVALID JSON ERROR

**Most likely root cause chain:**
1. Streaming sends JSON incrementally via `astream_events()`
   - Chunk 0: `{"quizzes": [`
   - Chunk 1: `{"question": "Q1",...`  (partial - missing fields)
   - Chunk 2: (more fields)
   - etc.

2. JsonOutputParser tries to parse at some point
   - Receives incomplete JSON: `{"quizzes": [...` (no closing)
   - Regex in JsonOutputParser fails
   - Raises OutputParsingError/exception

3. Exception caught in quiz.py except block (line 157)
   - Calls llm_chain.invoke() to get raw content
   - But raw_content from streaming is PARTIAL
   - _extract_quiz_json_payload(partial_json) returns NONE
   - Condition "if not repaired" is True
   - Re-raises exception → OUTPUT_PARSING_FAILURE

4. Error propagates to user
   - No recovery attempted
   - Chat returns error response with no metadata

**Why it happens:**
- JsonOutputParser expects COMPLETE JSON
- Streaming sends INCOMPLETE JSON to parser
- Fallback parser designed for text extraction, not streaming repair
- No streaming-aware JSON parser/accumulator exists

**Evidence:**
- Works with streaming=False in supervisor (line 35 in llm_model.py)
- Fails with streaming=True for generation (line 17)
- No special handling for streaming JSON in quiz.py chain
